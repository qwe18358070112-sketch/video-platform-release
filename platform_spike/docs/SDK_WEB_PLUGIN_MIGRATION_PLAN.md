# SDK / Web 插件迁移方案

## 目标

把当前项目从“桌面客户端自动化”迁移成“平台能力驱动的受控预览运行时”。

迁移后不再依赖：

- 宫格截图识别
- 全屏/窗口识别
- 面板点击定位
- 当前客户端界面状态猜测

## 官方资料结论

### 1. 产品定位

海康的“视频融合赋能平台”本身是平台型产品，不是只给人手动点的桌面客户端。

### 2. Web 插件路线可行

海康开放平台提供了“视频 WEB 插件开发指南”和较老的 “Web 3.2 控件开发包编程指南”，说明平台存在可嵌入页面的预览承载能力。

### 3. SDK 路线可行

HCNetSDK 文档里可以看到 `NET_DVR_CLIENTINFO` 这类以窗口句柄承载预览的结构；同时文档也明确说明即使不显示也可以继续取流。

### 4. 当前 UIA 路线不通

本仓库的原生 `FlaUI.UIA3` 探针已经验证：

- `窗口分割` 读不到
- `全屏/退出全屏` 读不到
- 布局项读不到
- `VSClient` 渲染窗存在，但没有稳定 UIA 信息

所以不应继续推进客户端 UI 自动化重构。

## 建议的新架构

```text
平台资源层
-> 预览后端层
-> 宫格编排层
-> 轮询状态机层
-> 运维/人工控制层
```

## 一、平台资源层

职责：

- 获取组织/区域/设备/通道清单
- 把“轮询对象”统一成内部资源模型

建议契约：

- `IResourceCatalog`
- `CameraResource`
- `ChannelResource`

输出示例：

```json
{
  "resourceId": "camera-34020000001320000001",
  "name": "大厅东门",
  "backend": "hik-platform",
  "previewToken": "...",
  "channelId": "..."
}
```

## 二、预览后端层

这一层专门隔离“到底是 SDK 还是 Web 插件”。

建议接口：

- `IPreviewBackend`
- `OpenAsync`
- `CloseAsync`
- `AttachAsync(tileHandle, resource)`
- `DetachAsync(tileId)`
- `EnterFullscreenAsync(tileId)`
- `ExitFullscreenAsync()`
- `SetLayoutAsync(layoutPreset)`

### 方案 A：Web 插件后端

适合前提：

- 平台已有插件接入方式
- 平台方更容易提供 token、地址、组织和设备查询接口

优点：

- 对平台资源模型更友好
- 页面化承载更容易做控制台和日志
- 后续可直接接 WebView2 或浏览器壳

风险：

- 插件适配浏览器/容器
- 安装和升级插件服务
- 插件生命周期管理

### 方案 B：HCNetSDK 后端

适合前提：

- 已掌握设备直连方式
- 能拿到设备地址、端口、用户名、密码或平台中转能力

优点：

- 可直接把预览绑定到自有窗格句柄
- 宫格和全屏状态完全由你自己的程序控制

风险：

- 设备登录和连接管理更重
- 平台资源和设备资源之间要做映射
- 多设备并发、断流恢复、权限控制都要自己补

## 三、宫格编排层

不再“识别当前是几宫格”，而是程序自己持有当前布局。

建议模型：

- `LayoutPreset = Four | Six | Nine | Twelve`
- `TileSlot`
- `LayoutSession`

核心规则：

- UI 上显示几宫格，由程序自己设定
- 放大/返回不再靠外部客户端界面推断
- 选中、放大、切回，全部变成内部状态转移

## 四、轮询状态机层

当前 `scheduler.py` 的动作语义仍然能保留，但状态真相来源要替换。

保留：

- 单击选中
- 放大停留
- 返回停留
- 下一路
- 暂停/继续
- 手动锁定

替换：

- `observed_mode`
- `observed_layout`
- `view=GRID/ZOOMED`

这些不再来自截图识别，而是来自：

- 预览后端返回值
- 当前布局会话
- 当前选中 tile
- 当前 fullscreen owner

## 五、最小实施顺序

### Phase 1

只做资源发现和受控预览，不做轮询。

验收：

- 能列出资源
- 能打开 4 宫格
- 能切 9 宫格
- 能把某一路放大
- 能退出放大

### Phase 2

接入轮询状态机。

验收：

- 固定 4 宫格轮询
- 固定 9 宫格轮询
- 手动暂停/恢复
- 手动切布局后继续轮询

### Phase 3

接入异常恢复与运维能力。

验收：

- 断流重连
- 插件/SDK 异常恢复
- 运行日志
- 人工接管

## 六、下一步建议

下一步不要继续修现有 `scheduler.py` 的布局识别。

应该做：

1. 跟平台方确认可用的 OpenAPI / SDK / 插件清单
2. 确认平台资源查询和预览鉴权方式
3. 先做一个“打开 4 宫格并切 9 宫格”的受控预览 POC
4. POC 跑通后再迁移轮询状态机
