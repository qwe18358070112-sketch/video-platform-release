# 根因与技术替换建议

## 结论

当前项目的主技术路线不适合作为这类视频客户端自动化的长期方案。

现状是：

- 运行态依赖 `pywin32 + pywinauto + keyboard`
- 识别依赖 `opencv + Pillow`
- 运行入口依赖 WSL 与 Windows 之间的同步桥接

这意味着程序在关键场景下，实际上是在用两类不稳定信号拼凑真相：

1. UIA 能读到的少量宿主控件
2. 视频画面本身的像素结构、分隔线、纹理强弱

而视频客户端的真正预览区/全屏区，往往是自绘或 GPU 渲染面。只要这部分没有稳定暴露成可自动化控件，继续堆视觉启发式就会持续出现：

- 宫格识别漂移
- 全屏/窗口识别错乱
- 动作发出后缺少可靠回读
- 暂停/恢复/人工切换后的状态衔接失败

## 不建议继续作为主路线的技术

### 1. 当前 Python + UIA + OpenCV 启发式

这套方案适合“普通桌面表单 + 少量点击”，不适合“视频墙 + 自绘渲染面 + 运行态闭环”。

它可以保留为临时兼容层，但不应该继续作为最终核心架构。

### 2. WinAppDriver 作为新主方案

不建议把 WinAppDriver / Appium Windows Driver 作为新的主路线。

原因：

- WebDriver 风格对原生桌面客户端并不总是最稳
- 生态上仍依赖 WinAppDriver server
- 维护状态不理想，后续仍可能卡在平台问题而不是业务问题

如果必须兼容 WebDriver 协议，可以只把它当过渡层，而不是最终核心。

## 建议的根治路线

### 路线 A：优先级最高

如果业务允许，不再自动化现有桌面客户端，而是直接改成“平台接口/SDK 驱动”的方案。

适用场景：

- 客户端背后是海康视频融合赋能平台或同类平台
- 平台有开放接口、Web 插件、SDK、预览组件或流媒体能力

目标：

- 自己做一个受控的预览/轮询窗口
- 布局状态由程序自己持有，不再靠截图推断
- 放大、返回、轮询顺序、停留时长都在自有界面里控制

这样可以彻底去掉：

- 宫格识别
- 全屏识别
- 面板点击定位
- 运行态猜测

### 路线 B：必须继续操作现有客户端时

把运行时核心迁移成原生 Windows 自动化，不再用当前的 WSL 同步桥作为主执行面。

建议技术栈：

- `C# / .NET 8`
- `FlaUI + UIA3`
- `Inspect.exe / FlaUInspect` 做现场自动化树验证

目标：

1. 先验证“窗口分割”“全屏”“退出全屏”“收藏夹/搜索”等控件是否能稳定枚举
2. 能枚举的控件全部走 UIA selector，不再走图像猜测
3. 只有对真正不可见的区域，才保留视觉兜底

### 路线 C：视觉层只做兜底，不再做主真相来源

如果某些按钮或状态完全无法通过 UIA/SDK 获取，可以保留图像自动化，但只允许它处理：

- 固定图标
- 固定文字
- 固定按钮

不允许它继续承担：

- 通过视频纹理猜宫格
- 通过动态画面猜放大/返回
- 通过几何结构猜当前真实模式

更适合做兜底的方向：

- Airtest 这类图像自动化框架
- OCR（识别中文菜单/按钮）
- 轻量目标检测模型（只识别固定图标或菜单状态）

## 推荐实施顺序

### 第一阶段

冻结现有 Python 运行态启发式，不再继续大规模堆规则。

### 第二阶段

新增一个原生 Windows POC：

- 进程只跑在 Windows 上
- 先做“读取当前窗口 / 当前模式 / 当前布局面板状态”的探针
- 验证 UIA 树是否足够覆盖关键控件

### 第三阶段

如果 UIA 覆盖不够，立刻转平台 SDK / Web 插件路线，不再继续尝试把视频画面识别做成主状态源。

### 第四阶段

只在必要处保留视觉兜底，并把它约束为“识别固定控件”，而不是“识别视频内容”。

## 对当前仓库的直接建议

1. 不再把 [detector.py] 当成主状态判定层。
2. 不再把 [window_manager.py] 的宿主窗口 + 附属渲染面推断当成最终真相来源。
3. 不再让 [layout_switcher.py] 通过截图变化去确认布局是否真正生效。
4. 不再把 [windows_bridge.sh] 作为长期运行时架构。
5. 新建原生 Windows 运行时目录，先做 UIA 探针，再决定 SDK 或混合兜底路线。
