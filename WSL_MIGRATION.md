# WSL Migration

This repository can be developed, self-tested, and packaged inside WSL, but the live automation runtime still depends on Windows desktop APIs such as `pywin32`, UI Automation, global hotkeys, and `ImageGrab`.

## What works in WSL

- Creating a virtual environment
- Installing cross-platform dependencies from `requirements.txt`
- Running `python -m compileall .`
- Running `python self_test.py`
- Building `dist/video_platform_release.zip`
- Editing the code from the Linux filesystem for better I/O performance

## What still requires Windows

- `python app.py --run --mode auto`
- `python app.py --calibrate windowed`
- `python app.py --calibrate fullscreen`
- `python app.py --inspect-calibration windowed`
- `python app.py --inspect-calibration fullscreen`
- `python app.py --dump-favorites`
- `python app.py --switch-layout 4|6|9|12|13`

On WSL/Linux these runtime commands now fail fast with a clear message instead of crashing on `win32` imports.

## Copy the project into WSL

From Windows PowerShell:

```powershell
wsl.exe sh -lc 'mkdir -p ~/projects/video_platform_release'
wsl.exe sh -lc 'cp -a /mnt/d/video_platform_release_fixed/video_platform_release/. ~/projects/video_platform_release/'
```

If you want the WSL copy to replace an older copy, remove the target first or switch `cp` to `rsync` if it is installed.

## Install and verify inside WSL

```bash
cd ~/projects/video_platform_release
./install_deps.sh
python -m compileall .
./self_test.sh
./build_release.sh
```

## Trigger Windows runtime actions from WSL

Use the WSL bridge when you want Codex running in WSL to drive the real Windows client:

```bash
./windows_bridge.sh install-deps
./windows_bridge.sh self-test
./run_windows_runtime.sh --mode auto
./calibrate_windows.sh windowed
./calibrate_windows.sh fullscreen
./inspect_windows_calibration.sh windowed
./switch_layout_windows.sh 9
./dump_favorites_windows.sh
```

What the bridge does:

- syncs the WSL repo into a Windows work directory
- runs `windows_bridge.ps1` through `powershell.exe`
- executes the real Windows `.venv\Scripts\python.exe`
- copies `config.yaml`, `logs/`, `tmp/`, and `dist/` back into WSL after the run

Default Windows work directory:

- `/mnt/d/video_platform_release_windows_runtime`
- falls back to `/mnt/c/video_platform_release_windows_runtime` if `D:` is unavailable

Override it with:

```bash
export VIDEO_PLATFORM_WINDOWS_WORKDIR=/mnt/d/custom_video_platform_runtime
```

If the target client is elevated and you explicitly want the app's UAC relaunch behavior, enable:

```bash
export VIDEO_PLATFORM_ALLOW_AUTO_ELEVATE=1
```

By default the bridge passes `--no-auto-elevate` so Codex gets a deterministic failure instead of losing control to a detached elevated process.

## Run the live automation after migrating

Keep the source code in WSL if you want, but launch the live automation from Windows because the target client and its automation hooks are Windows-only. Two practical options:

1. Use the Windows copy of the release package generated by `build_release.py`.
2. Copy the updated files back to a Windows path and run the existing `.bat` entrypoints there.

## Recommended split

- WSL: coding, static validation, release packaging
- Windows: calibration, runtime automation, UIA inspection, hotkeys, screen capture
