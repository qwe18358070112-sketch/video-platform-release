# 旧项目目录与新交付目录的迁移指南

## 结论先说

建议你现在这样处理：

- **旧项目目录**：保留，作为备份和历史问题对照
- **新交付目录**：作为你和 Codex 后续唯一继续迭代的主目录

不要再直接在旧目录上继续追加修改。

---

## 一、为什么不建议继续在旧目录上做

旧目录通常会混着这些东西：

- 历史日志
- 临时截图
- 试验脚本
- 旧版文档
- 已经过时的提示词
- 不完整或不一致的配置

这会带来两个问题：

1. Codex 很容易被旧文件误导
2. 你自己也很难分辨“哪个版本才是当前有效版本”

---

## 二、现在最推荐的目录策略

### 目录 A：旧目录
用途：
- 保留原始历史
- 对照问题日志
- 对照老版本行为
- 发生新问题时回看历史证据

### 目录 B：新交付目录
用途：
- 当前唯一主开发目录
- 当前唯一主运行目录
- 当前唯一主打包目录
- 当前唯一让 Codex 接管的目录

---

## 三、哪些东西应该从旧目录迁移到新目录

### 建议迁移

1. `config.yaml` 中你已经现场验证过仍然有效的参数
   - 尤其是标定比例
   - 停留时间
   - 点击位置比例
   - 顺序配置

2. 有价值的历史日志
   - 能复现问题的日志
   - 能对照问题的截图
   - 能说明“旧版本为什么错”的视频

3. 真实收藏夹名称映射
   - `cell_labels`
   - 收藏夹顺序样本

### 不建议直接迁移

1. 历史临时脚本
2. 旧版 README / 旧版提示词
3. 不确定是否仍适用的批处理
4. 不知道来源的测试产物

---

## 四、最稳妥的迁移动作

### 方法一：人工迁移

1. 解压新的交付包到新目录
2. 只打开旧目录中的以下内容做对照：
   - `config.yaml`
   - `logs`
   - `screenshots`
   - 关键视频
3. 把确认仍有效的参数抄到新目录的 `config.yaml`
4. 运行：

```powershell
python self_test.py
```

5. 再做一轮短程实测

### 方法二：让 Codex 迁移

给 Codex 的规则应当是：

1. 旧目录只读，先不要直接改
2. 新目录作为唯一主工作目录
3. 只把旧目录中这些内容迁移进新目录：
   - 配置差异
   - 历史日志
   - 有价值截图
4. 迁移后必须重新跑：
   - `python -m compileall 项目根目录`
   - `python self_test.py`

---

## 五、迁移后第一轮必须复核什么

迁移完成后，最少复核下面五项：

1. 动作路径是否仍然是：
   - 单击选中 -> 双击放大 -> 停留 -> 双击返回 -> 停留 -> 下一路
2. F11 是否仍然重走当前路
3. F9 是否仍然可以连续步进
4. 标定图是否仍然正确
5. `runtime_guard` 是否启用并能留证截图

---

## 六、什么时候旧目录还能继续有用

只有三种场景：

1. 你要对比“旧版为什么会误点”
2. 你要拿旧日志给 Codex 做根因分析
3. 你要回退并提取某个仍然有效的配置值

除此之外，不建议把旧目录当成当前运行目录继续使用。
